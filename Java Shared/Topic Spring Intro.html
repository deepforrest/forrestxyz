Spring Frame Work:
Spring
  • Umbrella Term
    • Spring framework is broken into modules. Applications can choose what modules they need.
    • projects built on spring framework
  • Complementary to Java EE
  • Spring Solutions
    • Ability to decouple
    • tight vs loose coupling
      • Tight = Objects are built with their components (Doughnut comes with the filling)
      • Loose = Objects are built and components are added at runtime (Doughnut is built ~ Runtime adds the filling)
Spring Framework
  • Framework built to improve EJB (Enterprise Java Beans)
  • POJO Based ~ Interface Driven
  • Light weight
  • Unobtrusive => Doesn't make development harder
  • It supports Aspect Oriented Programming

Framework Design Philosophy
  • Provide choice at every level
  • Accommodate diverse perspectives. 
    • Flexible
    • Supports a wide range of application needs with different perspectives
  • Maintain strong backwards compatibility.
  • Care about API design. 
  • Set high standards for code quality.

How Does Spring Work:
Spring Beans
  • Spring Beans (not to be confused with Java Beans) are POJOs' that are registered with spring
    • registered with IOC Container
IOC Container
  • Inversion of Control 
  • Idea that control is given to spring and no longer in the hands of the developer
    • It does the heavy lifting
    • Injects dependencies for us!
IOC Container Goal:
IOC Container
It is responsible for instantiating, configuring and assembling the Beans (spring beans!)
We just need to give it instructions
Decouple
  • Execution from configuration
    • Telling spring that two things are related, but we will configure them at a later time
  • Makes app more modular
  • Achieves loose coupling
  • Business logic focused
  • Less time on code implementation
    • No mundane knowledge needed
    • Automates an allows logical implementation
IOC Container vs. Application Context::
Application Context
(IOC)
The IOC Container IS the Application Context!
  • It is provided by the spring context module
  • It is an interface (within Java) 
    • It is also a sub interface of Bean Factory
  • Earlier Versions of spring have Bean Factory as the IOC Container

  • Application Context extends Bean Factory
BF
  • Was a sophisticate implementation of the factory design pattern
    • Lazily & programmatically instantiated beans as a singleton
AC
  • Is an extension of BF. 
  • (Eagerly) Instantiates beans and is capable of scopes other than singleton.
Application Context (IOC):
What the AC does...
  • Provides dependence injection (Bean Writing!)
  • Manages the Bean (spring) lifespan
  • Managing Bean Scope
Dependency Injection
  • A process whereby objects define their dependencies; That is the objects they work with, only through constructor arguments, args to a factory method or properties that are set on the object instance after it is constructed or returned from a factory method.
  • The container then injects dependencies in a bean.
Bean Life-Cycle:
  1. Instantiates (spring) bean
  2. Populate bean properties
  3. Bean name aware  ~  .setBeanName()
  4. Bean factory aware  ~  .setBeanFactory()
  5. Bean post processor pre-initialization
  6. Initializing bean  ~  .afterPropertiesSet()
  7. (optional) Custom Init
  8. Bean post processor post initialization
  9. Bean ready for use!
  10. Destroy it  ~  DisposableBean.destroy() 
    • Optional - has a custom destroy method.
Bean Scope:
Bean Scope
(AC Scopes)
  • Scope tells us how many objects in Java correspond to a given bean definition
  • Singleton scope => Only a single bean object exists in the container
  • Prototype scope => Any number of bean objects can exist in the container
Annotations
  • Stereotype Annotations
    • Same function different names
    • Registers a POJO as a spring bean
  • @Component: Very General
  • @Service: Indicates the bean is part of the service layer
    • service layer == business layer
    • safe practices
  • @Repository: Indicates the bean is part of the repository layer
    • DAO
  • @Autowired: "Automagically" wires beans
    • Used on member variables
Autowired
vs
Inject
  • Autowired -- comes from spring //// Injected -- comes from Java API
  • They both extract something (data)
  • Matched byType. Resticts using @Qualifier and maches byName.
Annotation Best Practices
  • Explicitly name component 
    • Ex: @Component("class-name")
  • Avoid @Qualifier
  • Scan specified packages
