STLC Overview:
                                              
Requirement Analysis
  • Test team studies requirements from a testing point of view to identify the testable requirements.
  • The QA team may interact with various stakeholders to understand the requirements in detail.
  • Requirements could be either functional or non functional.
Activities
  • Identify the types of tests performed
  • Gather details about testing priorities and focus.
  • Prepare Requirement Tractability Matrix (RTM).
  • Identify test environment details where testing is supposed to be carried out.
  • Automation feasibility analysis (if required).
Documents
  • RTM
  • Automation feasibility report
Test Planning
  • Senior QA manager will determine effort and cost estimates for the project.
    • finalize the Test plan document.
  • Test strategy is determined.
Activities
  • Preparation of Test Plan document.
  • Test tools are selected.
  • Test effort estimation.
Documents
  • Test Plan
  • Effort Estimation 
Test Case Development
  • Create, Verify and rework test cases and test scripts.
  • Test data is identified or created then reviewed and reworked if needed.
Activities
  • Create test cases and automation scripts.
  • Review and baseline test cases and scripts.
  • Create test data.
Returns
  • Test cases/scripts.
  • Test data.
Test Environment Setup
  • Software and hardware conditions under which a work product is tested.
  • Can be done in parallel with Test Case Development phase.
  • If an environment is provided a smoke test is done.
Activities
  • Setup test Environment and test data.
  • Perform smoke test on the build.
Returns
  • Environment is set up.
  • Smoke test results.
Test Execution
  • Tests carry out the testing based on the test plans and test cases prepared.
  • Bugs are reported back to the development team.
  • Retesting as necessary.
Activities
  • Execute tests
  • Document test results and log defects for failed cases.
  • Map defects to test cases in RTM.
  • Retest defects once fixed.
  • Track the defects to closure.
Returns
  • Completed RTM with the execution status.
  • Test cases updated with results.
  • Defect reports.
Test Cycle Closure
  • Testing tea will meet, discuss and analyze testing artifacts to identify strategies that have to be implemented in the future.
    • Remove the process bottlenecks for future test cycles.
    • share best practices for future projects.
Activities
  • Evaluate cycle completion criteria based on Time, Test coverage, Cost, Software, Critical business objectives and quality.
  • Prepare test metrics.
  • Prepare Test Closure report.
  • Qualitative and quantitative reporting of quality of the work product to the client.
  • Test result analysis to find out the defect distribution by type and severity.
Documents
  • Test Closure report
  • Test Metrics

Unit Testing
  • Most micro level of testing
  • Test each component stand-alone (without dependencies)
  • Usually requires mocking (Mockito)
  • ex: JUnit, TestNG
Integration Testing
  • Test components with dependencies
  • Components work with each other correctly
  • Logically related models are tested as a group
  • ex: Selenium (SE)
Systems Testing
  • Testing entire application as a whole
  • Make sure entire app works without errors.
User Acceptance Testing (UAT)
  • Test that all requirements/user stories are met
  • Simulate an end user
  • Actually does all the stuff it is supposed to do 
  • Comes in two flavors:
    • Alpha testing
      • Done inhouse
    • Beta Testing
      • Done out of the house (outsourced)
Functional/Non-Functional Testing:
Functional vs Non-Functional 
  • Functional: What the app does
  • Non-Functional: How well the app performs.
