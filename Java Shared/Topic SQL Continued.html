Notes in Depth
Transaction:
What is a Transaction
  • A unit of work performed against the database.
  • It is the population of one or more changes to the database.
  • Group multiple operations together and execute at once.
ACID
  • Atomicity, Consistency, Isolation, Durability
  • Atomicity:
    • Operations either occur successfully or not at all.
    • Transactions will be aborted and changes will be rolled back.
  • Consistency:
    • Insures that the database properly changes states on a successful transmitted transaction.
    • No transaction should have any adverse effect on the data residing in the database.
    • Referential Integrity/Constraints are maintained.
  • Isolation
    •  Enables transactions to operate independently of each other.
    • How strict with transactions operations at the same time are'.
  • Durability
    • The result or effect of a committed transaction persists in case of system failure.
Transaction Isolation Levels:
What are Transaction Isolation Levels
  • The degree to which two transactions will interact with each other over the same data.
  • As our app becomes more complex, we must account for transactions that occur at the same time.
  • The Higher the isolation level, the more careful the system is about avoiding conflict.
    • The locking overhead can increase as concurrency decreases
  • As isolation level increases efficiency decreases
Isolation Level
  1. Read Uncommitted
  2. Read Committed
  3. Repeatable Read
  4. Serializable 
Phenomena 
  • Transaction problems
    • Dirty Read
    • Non Repeatable Reads
    • Phantom Reads

Normalization:
What is Normalization
  • Database normalization is a design pattern for databases that aims to reduce duplicate data/redundancy
  • Aims to insure referential integrity
  • Atomic data - Data that has been broken down into smallest possible unit 
Levels of Normalization
0NF: No normalization
1NF: First normal form
  • All data must be atomic
    • Data in each cell is micro as possible
    • Should have a unique identifier (PK)
2NF: Second normal form
  • First normal form + No partial dependencies
    • All values must be identified by a single column
    • No composite key
3NF: Third normal form
  • Second normal form + no transit dependencies
    • No columns are dependent on a column that is not the PK
